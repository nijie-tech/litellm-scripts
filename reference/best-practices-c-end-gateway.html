C 端 Agent 的 LiteLLM Gateway 配置最佳实践

场景:自研 C 端 agent,LiteLLM 作 AI gateway,产品方统一付费、单实例起步。 这是长期回看用的浓缩卡。配套讲解见 第 06 课。 口诀仍是四抽屉:model_list / litellm_settings / general_settings / router_settings。

0 · 黄金法则

  • 配置进 config.yaml,密钥进环境变量。密钥一律 os.environ/KEY,绝不写死。
  • 有状态的东西全部外置。虚拟 key/花费 → Postgres;缓存/全局限流/多实例路由 → Redis。无状态进程才能横向扩展。
  • C 端要"快失败"。消费者等不了——超时短、重试少、缓存多、fallback 稳,优先保住"秒回"和"不报错"。
  • 一定带 user 字段。产品方付费下,不传 end-user 标识 = 无法按用户防滥用、出事查不到人。

1 · 生产基线 config.yaml(带注释 · 可直接改用)

这是把下面四个维度拼成的一份完整范本。先理解整体,再看分维度细则。[1]

model_list:
  # —— 主力:OpenAI(可挂多 key 做负载均衡:同名多条目) ——
  - model_name: chat-default
    litellm_params:
      model: openai/gpt-4o
      api_key: os.environ/OPENAI_API_KEY
      rpm: 500
  # —— 备用/降本:国内模型,与主力互为 fallback ——
  - model_name: chat-cn
    litellm_params:
      model: openai/deepseek-chat
      api_base: https://api.deepseek.com
      api_key: os.environ/DEEPSEEK_API_KEY

litellm_settings:
  # 可靠性
  num_retries: 2                 # C 端:少重试,快失败
  request_timeout: 60            # 单请求上限(秒)
  fallbacks: [{"chat-default": ["chat-cn"]}, {"chat-cn": ["chat-default"]}]  # 互备
  context_window_fallbacks: [{"chat-cn": ["chat-default"]}]
  enable_pre_call_checks: true
  # 成本护栏(全局兜底,防单个 end-user 烧穿)
  max_end_user_budget: 5         # 每个 end-user 累计花费上限(USD)
  # 可观测性
  callbacks: ["prometheus"]      # /metrics 大盘;再按需加 "langfuse"
  json_logs: true                # 结构化日志
  set_verbose: false             # 生产关闭啰嗦日志
  # 缓存(省钱降延迟)——注意用 host/port,不要 redis_url
  cache: true
  cache_params:
    type: redis
    host: os.environ/REDIS_HOST
    port: os.environ/REDIS_PORT
    password: os.environ/REDIS_PASSWORD
    ttl: 600
  # 隐私:第三方追踪只记花费不记内容(C 端合规)
  turn_off_message_logging: true

general_settings:
  master_key: os.environ/LITELLM_MASTER_KEY    # sk- 开头
  database_url: os.environ/DATABASE_URL        # 虚拟 key/花费落库
  alerting: ["slack"]                          # 预算/异常告警
  proxy_batch_write_at: 60                     # 花费批量写库,降 DB 压力
  database_connection_pool_limit: 10
  allow_requests_on_db_unavailable: true       # DB 抖动时仍能服务(C 端不掉线)

router_settings:
  routing_strategy: simple-shuffle             # 起步够用;延迟敏感可换 latency-based-routing
  # 多实例时再开(单实例可省):
  # redis_host: os.environ/REDIS_HOST
  # redis_port: os.environ/REDIS_PORT
  # redis_password: os.environ/REDIS_PASSWORD

guardrails:
  - guardrail_name: pii-mask
    litellm_params:
      guardrail: presidio        # 入站前对 PII 脱敏
      mode: pre_call
      pii_entities_config:
        EMAIL_ADDRESS: MASK
        CREDIT_CARD: MASK
        PHONE_NUMBER: MASK

必备环境变量

export LITELLM_MODE=PRODUCTION          # 关掉 load_dotenv 等开发行为
export LITELLM_LOG=ERROR                # 抑制框架 info 日志
export LITELLM_MASTER_KEY=sk-...        # 管理总钥匙
export LITELLM_SALT_KEY=sk-...          # 加密落库的厂商凭证,生产必设且勿改
export DATABASE_URL=postgresql://...
export REDIS_HOST=... REDIS_PORT=6379 REDIS_PASSWORD=...
export OPENAI_API_KEY=... DEEPSEEK_API_KEY=...

2 · 分维度最佳实践

维度

成本控制 & 防滥用(产品方付费)

做法怎么配为什么
按 end-user 跟踪每次请求体带 "user": "<你的用户ID>"不传就无法按用户算账/限额/封禁[2]
全局兜底预算litellm_settings.max_end_user_budget任一用户烧穿即被拦(401),防刷爆
分层套餐限额/budget/new(设 tpm/预算)→ /customer/new 绑定 user_id免费/付费用户给不同 budget_id[2]
超额告警general_settings.alerting: ["slack"]预算/异常实时通知,别等账单才发现
降本路由便宜模型作主力,贵模型作 fallback;开缓存多数 C 端请求重复度高,缓存直接省钱
反模式:所有 C 端用户共用一把虚拟 key 且不传 user。结果=无法定位滥用者、无法分层限额、被刷爆只能整体关停。
维度

延迟 & 流式体验

做法怎么配为什么
流式透传请求带 "stream": true(OpenAI 兼容,网关原样透传 SSE)首 token 尽快出,C 端体感快
快失败num_retries: 1~2 + 适中 request_timeout重试堆叠会让用户干等;宁可早 fallback
延迟路由多部署时 routing_strategy: latency-based-routing自动避开当前慢的部署
缓存cache: true + redis命中=毫秒级返回且零成本
维度

可靠性容灾

做法怎么配为什么
跨厂商互备fallbacks 双向:OpenAI↔国内模型任一家限流/宕机,业务无感切换[3]
超长上下文兜底context_window_fallbacks + enable_pre_call_checksprompt 超窗自动转大窗口模型,不报错
DB 抖动不掉线allow_requests_on_db_unavailable: trueDB 短暂不可用时仍放行请求[1]
健康检查给 LB/探针挂 /health/liveliness实例异常自动摘除
维度

内容安全护栏

做法怎么配为什么
PII 脱敏guardrails + guardrail: presidio, mode: pre_call入站前屏蔽邮箱/卡号等,防泄漏[4]
内容审核moderation / lakera / bedrock 等 guardrailC 端 UGC 合规,拦截违规输入输出
执行时机pre_call(查输入)/ post_call(查输出)/ during_call(并行不阻塞)按"拦输入还是拦输出"选时机[4]
不外泄内容turn_off_message_logging: true第三方监控只记花费/延迟,不记对话

3 · 通用生产基线(别漏)

项设置
生产模式LITELLM_MODE=PRODUCTION
日志LITELLM_LOG=ERROR + json_logs: true + set_verbose: false
凭证加密LITELLM_SALT_KEY 必设、设后勿改(改了已落库的厂商 key 解不开)
写库压力proxy_batch_write_at: 60 批量写花费
连接池database_connection_pool_limit: 10
镜像用非 root 官方镜像;K8s 下单 pod 单 worker、按内存横向扩展
Redis用 host/port/password,别用 redis_url(后者约慢 80 RPS)[1]

4 · 上线前检查清单

  • ☐ 所有密钥走 os.environ/,仓库里无明文
  • ☐ master_key + salt_key 已设为强随机且妥善保管
  • ☐ 请求链路确实带上了 user 字段(端到端验一次)
  • ☐ max_end_user_budget / 分层 budget 已设,并实测超额被拦(401)
  • ☐ 故意改坏主力 key,验证 fallback 切到备用模型成功
  • ☐ /metrics 能抓到 spend/latency;告警通道(Slack)实测能收到
  • ☐ 缓存命中头 x-litellm-cache-key 出现;/cache/ping 通
  • ☐ guardrail 实测能脱敏/拦截;turn_off_message_logging 已开
  • ☐ LITELLM_MODE=PRODUCTION + json_logs + 非 root 镜像
