C 端 Agent + LiteLLM Gateway 整体方案

基于你的实况:远程网关 + 8 模型(qwen/kimi/deepseek,DB 存储,已配价)+ 四档套餐。 核心原则:应用做决策,网关做执行与治理。

一、端到端请求流

用户(C 端) │ ▼ 你的 Agent 后端 ──────────────── 决策层(本应用) ① 取上下文:功能入口 feature、用户 user_id、本轮文本、会话已用模型 ② 选模型:FEATURE_MODEL 直选 → 否则中文复杂度打分分档(app_side_router_example.py) ③ 组请求:{ model: 选出的, user: user_id, messages, stream } │ Authorization: Bearer <本应用的虚拟 key> ▼ LiteLLM Gateway ─────────────── 执行+治理层(网关) a. 验虚拟 key、查该 key/该 user 预算与限速 → 不够直接 401 拦截 b. 按 model 路由到对应厂商部署 c. 失败/限流 → 按 fallbacks 自动切备用模型(应用无感) d. 算成本 → 记到 user_id 头上;打 /metrics、日志 ▼ qwen / kimi / deepseek 厂商 API(真实 key 只存网关,应用永不接触)

三、分层落地

① 接入层

  • Agent 后端只配一个 OPENAI_BASE_URL = 网关地址 + /v1 和一把专属虚拟 key(不是 master、不是 test)。
  • 厂商真实 key 只存网关(已是 DB credential:qwen-plan / kimi 等),应用永不持有。
  • 加/换模型 = 网关侧操作,应用代码不动。

② 选模型(应用)

  • 模块:scripts/app_side_router_example.py → 适配进你的框架。
  • 功能入口表 FEATURE_MODEL(代码→kimi-k2.7-code、翻译→deepseek-flash…)。
  • 通用对话:中文关键词+长度打分 → SIMPLE/MEDIUM/COMPLEX/REASONING → 模型。
  • 会话内不降档(对齐主流:稳定>聪明,省缓存)。
  • 关键词/阈值都是你自己的代码,可持续调,无英文黑盒。

③ 终端用户治理(网关)

  • 每个请求必须带 "user": "<user_id>"。
  • 新用户登记并绑档:./customers.sh add <user_id> --tier free。
  • 四档套餐已建(free/pro/elite/ultra),金额单位=人民币。
  • 升级/降级 = 改绑档位;滥用 = customers.sh block。

④ 可用性兜底(网关)

litellm_settings:
  num_retries: 2
  fallbacks: [
    {"deepseek/deepseek-v4-flash": ["qwen3.7-plus"]},
    {"qwen3.7-plus": ["deepseek/deepseek-v4-pro"]},
    {"moonshot/kimi-k2.6": ["qwen3.7-max"]},
    {"qwen3.7-max": ["moonshot/kimi-k2.6"]}
  ]
