LiteLLM Proxy 网关 · 第 01 课

启动你的第一个 Proxy:一份配置,接通 OpenAI 与国内模型

这一课服务于你的使命:生产级网关把所有模型收敛到一个 OpenAI 兼容入口。 今天的目标很小但是地基——写出最小的 config.yaml,让一个进程同时代理 OpenAI 和一个国内模型,并用同一套接口调通。后面的虚拟 key、fallback、监控,全都长在这份配置之上。

一、心智模型:config.yaml 的四个抽屉

Proxy 的全部行为由一份 config.yaml 决定。它只有四个顶层段,把它们想成四个抽屉, 你以后查任何配置都先问"这属于哪个抽屉":[1]

  • model_list — 有哪些模型、怎么连。今天只碰这个抽屉。
  • litellm_settings — 全局行为:日志、缓存、超时、回调。
  • general_settings — 网关自身:master_key、数据库、预算、限流。
  • router_settings — 路由策略:负载均衡、fallback、重试。

顺序记忆法:模型 → 全局 → 网关 → 路由。今天只开第一个抽屉,其余三个是后续课程。

二、关键洞察:model_name 是"对外名",model 是"真实串"

核心抽象 model_list 里每一项都有两层名字:
• model_name —— 客户端调用时写的名字(你随便起,对外稳定)。
• litellm_params.model —— LiteLLM 真正去调的串,格式是 提供商/模型,比如 openai/gpt-4o。
这层间接正是网关的价值:应用永远只认 model_name; 底层换厂商、换模型、加备用,只改配置,应用代码一行不动。[2]

国内模型怎么接?——"OpenAI 兼容"通法

绝大多数国内模型(DeepSeek、通义、智谱等)都提供 OpenAI 兼容接口。 通法是:model 写成 openai/<模型> 走 OpenAI 协议,再用 api_base 把请求指到厂商的地址。api_key 用 os.environ/变量名 从环境变量读,绝不写死在文件里。[2]

三、动手:四步跑通

  1. 安装带 Proxy 的 LiteLLM(注意是 [proxy] 这个 extra,不是 brew): 
    pip install 'litellm[proxy]'
  2. 把密钥放进环境变量(以 DeepSeek 作国内模型示例): 
    export OPENAI_API_KEY="sk-你的openai key"
export DEEPSEEK_API_KEY="你的deepseek key"
  3. 写最小 config.yaml:

    config.yaml

    model_list:
  # —— OpenAI ——
  - model_name: gpt-4o            # 对外名,客户端就写这个
    litellm_params:
      model: openai/gpt-4o        # 真实串:提供商/模型
      api_key: os.environ/OPENAI_API_KEY

  # —— 国内模型(OpenAI 兼容 + 自定义 api_base) ——
  - model_name: deepseek-chat
    litellm_params:
      model: openai/deepseek-chat
      api_base: https://api.deepseek.com
      api_key: os.environ/DEEPSEEK_API_KEY
  4. 启动网关,默认监听 http://0.0.0.0:4000:[3] 
    litellm --config config.yaml

验证:用 OpenAI 兼容接口调它

同一个 endpoint,只靠 model 字段切换底层厂商——这就是网关的全部魔力:

curl http://0.0.0.0:4000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "deepseek-chat",
    "messages": [{"role": "user", "content": "用一句话介绍你自己"}]
  }'

把 "model" 改成 "gpt-4o",请求就转去了 OpenAI——客户端代码完全没变。 这正是你将来给团队提供的体验。
