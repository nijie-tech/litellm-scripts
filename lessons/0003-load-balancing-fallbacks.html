LiteLLM Proxy 网关 · 第 03 课

负载均衡与 fallback:让网关在故障下不掉线

这一课服务于你的使命:生产网关的命脉是"某个模型限流/挂了,业务不能跟着挂"。 今天开 router_settings 抽屉,学会两件事:把同一个模型名摊到多个部署上分摊压力(负载均衡), 以及当首选模型失败时自动切到备用模型(fallback)——比如 DeepSeek 限流就兜底到 OpenAI。

一、负载均衡:同名,多部署

还记得第 01 课的间接关系吗?model_name 是对外名。今天把它推到极致: 多个 model_list 条目共用同一个 model_name,LiteLLM 就会把发往这个名字的流量 分摊到这些部署上。[1]

核心抽象 一个 model_name = 一个逻辑模型;它背后可以挂 N 个物理部署 (不同 key、不同区域、甚至不同厂商)。客户端只认那个名字,扩容/容灾全在配置里完成。

config.yaml — 两个 key 摊到同一个逻辑名

model_list:
  - model_name: gpt-4o                 # 逻辑名
    litellm_params:
      model: openai/gpt-4o
      api_key: os.environ/OPENAI_KEY_A
      rpm: 100                         # 这个部署的配额
  - model_name: gpt-4o                 # 同名!第二个部署
    litellm_params:
      model: openai/gpt-4o
      api_key: os.environ/OPENAI_KEY_B
      rpm: 100

选择路由策略

流量怎么摊,由 router_settings.routing_strategy 决定:[1]

  • simple-shuffle — 默认,按权重随机。够用、最省心。
  • least-busy — 发给当前在途请求最少的部署。
  • latency-based-routing — 发给近期最快的部署。
  • usage-based-routing — 按已用 token 量避开快触顶的部署。
router_settings:
  routing_strategy: simple-shuffle

二、重试 vs fallback:两种不同的"再来一次"

别混淆 • retry(重试) = 同一个逻辑模型再试几次(会换到同名的其它部署)。治"偶发抖动"。
• fallback(兜底) = 重试都失败后,切到另一个模型名。治"这个模型整体不可用"。
顺序:先在 gpt-4o 的各部署间 num_retries 次重试 → 还不行 → 按 fallback 切到备用模型。

两者都写在 litellm_settings 里。fallbacks 是一个"主模型 → 备用模型列表"的映射:[1]

config.yaml — 重试 + 兜底

litellm_settings:
  num_retries: 3                 # 每个逻辑模型失败前重试次数
  request_timeout: 30            # 单请求超时(秒)
  fallbacks: [{"gpt-4o": ["deepseek-chat"]}]   # gpt-4o 不行就切 deepseek-chat
  # 提示词超长则换大窗口模型(需开 enable_pre_call_checks)
  context_window_fallbacks: [{"deepseek-chat": ["gpt-4o"]}]
  enable_pre_call_checks: true

生产实用搭配:OpenAI 与国内模型互为 fallback。任意一家限流/抖动,业务自动走另一家, 用户无感。这正是你使命里"可靠性"的落地形态。

三、动手验证 fallback 真的生效

  1. 把上面的 fallbacks 写进 config,故意把 gpt-4o 的 api_key 改成一个无效串。
  2. 重启网关,正常请求 "model":"gpt-4o"。
  3. 观察:gpt-4o 调用失败 → 自动切到 deepseek-chat 并成功返回。看网关日志能看到这次切换。
curl http://0.0.0.0:4000/v1/chat/completions \
  -H "Authorization: Bearer $LITELLM_MASTER_KEY" \
  -H "Content-Type: application/json" \
  -d '{"model":"gpt-4o","messages":[{"role":"user","content":"hi"}]}'
# 期望:尽管 gpt-4o 的 key 是坏的,仍拿到回复(来自 deepseek-chat)
