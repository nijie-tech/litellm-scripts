LiteLLM Proxy 网关 · 第 02 课

master_key 与虚拟 key:给每个应用发独立凭证、设预算与限流

这一课服务于你的使命:生产网关不能让所有应用共用一把真厂商 key。 今天你给网关装上"前台":一把管理总钥匙(master_key),再由它签发 一次性的虚拟 key——每个应用一把,各自限预算、限速率、限可用模型。某个应用泄漏或超支, 吊销那一把即可,真厂商 key 永不出网关。

一、为什么需要虚拟 key

第 01 课里,任何人只要够到 :4000 就能白嫖你的 OpenAI 额度——网关此时是"裸奔"的。 生产环境要的是:一应用一凭证、可计费、可限额、可吊销。LiteLLM 用两层凭证解决:

两层凭证 • master_key —— 管理总钥匙,必须以 sk- 开头。只有它能签发/管理虚拟 key。[1]
• virtual key —— 由 master_key 通过 /key/generate 现签现发,带预算/限速/模型白名单,发给具体应用当 Bearer token 用。
类比:master_key 是总台保险柜钥匙,虚拟 key 是前台发给每位客人的房卡——能单独失效,丢了不影响保险柜。

二、关键前提:虚拟 key 需要数据库

这是今天最容易踩的坑:master_key 单独就能用(保护网关),但只要你想签发并存储 虚拟 key、记录它们的花费,就必须接一个 PostgreSQL,通过 DATABASE_URL 提供。 网关启动时会自动建表。[1]

记忆钩子:master_key 在内存里就能验,虚拟 key 的预算/花费必须落库。没有数据库 = 没有虚拟 key。

三、动手:三步发出第一把房卡

  1. 起一个 Postgres(本地用 Docker 最快): 
    docker run -d --name litellm-db \
  -e POSTGRES_PASSWORD=litellm \
  -e POSTGRES_DB=litellm \
  -p 5432:5432 postgres:16
  2. 在 config.yaml 里开 general_settings 抽屉,并把两个密钥放进环境变量:

    config.yaml(在第 01 课基础上追加)

    general_settings:
  master_key: os.environ/LITELLM_MASTER_KEY
  database_url: os.environ/DATABASE_URL
    export LITELLM_MASTER_KEY="sk-master-自己起一个长随机串"
export DATABASE_URL="postgresql://postgres:litellm@localhost:5432/litellm"
litellm --config config.yaml
  3. 用 master_key 签发一把虚拟 key——只给某应用 deepseek-chat、月预算 $10、限速: 
    curl http://0.0.0.0:4000/key/generate \
  -H "Authorization: Bearer $LITELLM_MASTER_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "models": ["deepseek-chat"],
    "max_budget": 10,
    "rpm_limit": 60,
    "duration": "30d",
    "metadata": {"app": "demo-app"}
  }'
    返回里会有一把 "key": "sk-..." ——这就是发给应用的房卡。[1]

验证:用房卡调用(而不是 master_key)

curl http://0.0.0.0:4000/v1/chat/completions \
  -H "Authorization: Bearer sk-刚才返回的虚拟key" \
  -H "Content-Type: application/json" \
  -d '{"model":"deepseek-chat","messages":[{"role":"user","content":"hi"}]}'

试试两件事看护栏生效:① 把 "model" 换成 "gpt-4o"——会被拒,因为这把 key 的白名单里没有它。 ② 反复快打超过 rpm_limit——会触发限速。花费会自动累加进数据库。
