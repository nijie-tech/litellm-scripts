C 端 Agent Gateway 的配置决策框架

这一课服务于你的使命:你已掌握机制,现在要的是"对 C 端 agent 该怎么配、为什么"。 本课不再讲机制,而是给你一套决策框架:从 C 端产品的四个约束出发,推出每个配置块的取舍。 配套的浓缩成果是 → 最佳实践参考卡(长期回看用,本课结束请收藏)。

一、先认清:C 端 agent 给网关的四条约束

所有最佳实践都从这四条推出来。记住约束,配置自然就有答案:

四条约束 → 配置取向 ① 你统一付费 → 必须按 end-user 算账、限额、防滥用。
② 消费者没耐心 → 快失败 + 流式 + 缓存,延迟优先于"多试几次"。
③ 用户会因报错流失 → 跨厂商 fallback、DB 抖动不掉线。
④ 面向公众要合规 → guardrails 脱敏/审核 + 不外泄对话内容。

二、逐块决策:为什么这么设

约束① 付费 → 成本与防滥用

关键动作只有一个却最容易漏:每次请求带 "user": "<你的用户ID>"。 LiteLLM 用它按终端用户累计花费,无需给每个 C 端用户单独建 key。[1] 在此之上:全局 max_end_user_budget 兜底防烧穿;免费/付费用户用 /budget/new + /customer/new 给不同套餐。 

// 你的 agent 调网关时,务必透传终端用户标识
{"model":"chat-default","user":"u_88231","messages":[...]}

虚拟 key vs end-user 的分工:虚拟 key 是给"你这个 agent 应用"的一把凭证; user 字段区分这把 key 背后的千千万万 C 端个人。两者叠加用。

约束② 没耐心 → 延迟与流式

C 端的反直觉点:重试不是越多越好。重试会让用户干等, 所以 num_retries 设小(1~2),配合较短 request_timeout,失败就快速 fallback。 流式 "stream": true 原样透传让首 token 早出;高重复请求交给缓存毫秒级返回。

约束③ 怕流失 → 可靠性

让 OpenAI 与国内模型双向互为 fallback:任一家限流/抖动,另一家顶上,用户无感。[2] 再加 allow_requests_on_db_unavailable: true——DB 短暂抖动时网关仍放行请求,不会因为记账库挂了而拒绝服务。[3]

约束④ 要合规 → 内容护栏

guardrails 段挂 presidio 在 pre_call 把邮箱/卡号等 PII 脱敏;按需加 moderation 审违规。 时机记法:pre_call 拦输入,post_call 查输出,during_call 并行不阻塞。[4] 再用 turn_off_message_logging: true 保证监控只记花费、不外泄对话。

三、两个一定要知道的"坑"

  • Redis 别用 redis_url。生产文档明确:用 host/port/password 分开写,redis_url 约慢 80 RPS。 (这纠正了第 05 课的写法。)[3]
  • LITELLM_SALT_KEY 设了就别改。它用来加密落库的厂商凭证;改掉后已存的 key 解不开。[3]
