LiteLLM Proxy 网关 · 第 05 课

缓存:省钱、降延迟,重复请求秒回

这一课服务于你的使命:生产网关里大量请求是重复的(同样的提示、同样的检索片段)。 每次都打厂商 API 既慢又烧钱。今天给网关接上缓存:相同请求直接返回上次结果, 并在第 04 课的监控里亲眼看到花费曲线下降、延迟变低。

一、心智模型:缓存是问厂商之前先翻的"账本"

开启后,网关在调用厂商前先查缓存:命中就直接返回存好的回复,完全不打厂商 API—— 省掉那次 token 费用,延迟从"秒级"降到"毫秒级"。[1] 缓存配置在 litellm_settings 抽屉: 

litellm_settings:
  cache: true
  cache_params:
    type: redis      # 生产首选;也可 local / s3 / qdrant-semantic
    ttl: 600         # 缓存存活 600 秒
    namespace: my-gateway   # 可选:key 前缀,多应用隔离

二、关键洞察:缓存要"共享"才在生产有意义

为什么用 Redis 而不是 local type: local 是单进程内存缓存——网关一重启就清空,多实例之间也各缓存各的,命中率低。 生产用 type: redis 把缓存放进外部共享存储:重启不丢、所有网关实例共用一份。 这和第 02 课"虚拟 key 要落 Postgres"是同一个道理——状态要外置才能横向扩展。

连 Redis 既可写在 cache_params 里,也可用环境变量:[1]

export REDIS_URL="redis://:你的密码@localhost:6379"

三、关键洞察:精确缓存 vs 语义缓存

两种命中方式,别搞混 • 精确缓存(默认) —— 请求逐字相同才命中。差一个标点就 miss。适合"完全重复"的调用。
• 语义缓存(qdrant-semantic 等) —— 按语义相近命中。"今天天气如何"和"今天天气怎样"可视为同一问。 命中率高,但有"答非所问"风险,需要调相似度阈值。[1]
起步建议:先上精确缓存,稳了再评估语义缓存。

四、动手:三步验证命中

  1. 起一个 Redis: 
    docker run -d --name litellm-redis -p 6379:6379 redis:7
  2. 开缓存并重启网关: 
    export REDIS_URL="redis://localhost:6379"
litellm --config config.yaml   # config 里已写 cache: true / type: redis
    先确认连通:curl http://0.0.0.0:4000/cache/ping
  3. 同一请求发两次,看第二次的速度与命中头: 
    curl -i http://0.0.0.0:4000/v1/chat/completions \
  -H "Authorization: Bearer $LITELLM_MASTER_KEY" \
  -H "Content-Type: application/json" \
  -d '{"model":"deepseek-chat","messages":[{"role":"user","content":"用一句话介绍缓存"}]}'
    第二次应明显更快,响应头里出现 x-litellm-cache-key 表示命中。[1]

回到第 04 课的 /metrics 或 Langfuse:命中的请求不产生厂商花费——你能直接看到 spend 不再增长。 这就是缓存价值的"可见证据"。

按请求微调

个别请求想跳过缓存或改 ttl,可在请求体里覆盖:

{"model":"deepseek-chat","cache":{"no-cache":true},"messages":[...]}  // 本次强制不读缓存
